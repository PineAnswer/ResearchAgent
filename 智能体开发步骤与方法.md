### 第一讲：智能体开发概述

#### 1.1 什么是智能体（Agent）？

**智能体（AI Agent）** 是一个能够感知环境、自主决策并采取行动以实现特定目标的人工智能系统。

简单类比：
- **普通大模型（LLM）** → 像一个顾问，你问它答，但它只能用已有的知识回答
- **智能体（Agent）** → 像一个助手，它不仅能回答问题，还能**使用工具**、**记忆对话**、**规划步骤**、**自主执行任务**

#### 1.2 智能体的核心组件

一个完整的智能体通常包含以下6大核心组件：

| 组件 | 作用 | 你的项目中的例子 |
|------|------|------------------|
| **大模型（LLM）** | 大脑，负责推理和决策 | ChatTongyi / DeepSeek |
| **工具（Tools）** | 手和脚，让智能体能做事 | 搜索工具、数据查询工具 |
| **记忆（Memory）** | 大脑的记忆，记住历史对话 | MemorySaver / AGENTS.md |
| **规划（Planning）** | 思考步骤，拆解任务 | LangGraph 状态图 |
| **技能（Skills）** | 专业能力，行为指南 | SKILL.md 文件 |
| **状态（State）** | 当前进度，上下文数据 | MessagesState / 自定义状态 |

#### 1.3 智能体的发展历程

```
第一代：基础LLM
  ↓
第二代：Chain（链）
  - 提示词 + 模型 = 固定流程
  - 例子：翻译链、摘要链
  ↓
第三代：ReAct Agent（反应式智能体）
  - 思考（Thought）→ 行动（Action）→ 观察（Observation）循环
  - 能调用工具，但流程较简单
  ↓
第四代：LangGraph 智能体
  - 状态图驱动，灵活的控制流
  - 支持分支、循环、条件判断
  - 你项目中的 chat.py 就是这种
  ↓
第五代：Deep Agents / 多智能体系统
  - 上下文工程、子智能体、人机协同
  - 你项目中的 FIFA 世界杯系统就是这种
```

这是智能体开发的基础。在构建智能体之前，你需要先掌握LangChain的三大核心概念。

#### 3.1 模型（Model）

模型是智能体的"大脑"，负责推理和生成。

**消息类型（Messages）：**

```python
from langchain_core.messages import HumanMessage, SystemMessage

# 单轮对话
response = model.invoke([
    HumanMessage("你好！")

# 带系统提示的多轮对话
response = model.invoke([
    SystemMessage("你是一个专业的翻译官"),
    HumanMessage("把'你好'翻译成英语"),
```

#### 3.3 链（Chain）

链是LangChain的核心概念，可以把多个组件串联起来。

**最基本的链：提示词 + 模型**
位置：[main.py](file:///f:/llm-hint/langchain-2025/web-project/langserve-api/main.py#L26)

```python
from langchain.prompts import ChatPromptTemplate
from langchain_community.chat_models.tongyi import ChatTongyi

# 定义组件
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个{role}"),
    ("user", "{question}")
])
model = ChatTongyi(streaming=True)

# 用 | 运算符串联（LCEL 语法）
chain = prompt | model

# 调用
result = chain.invoke({
    "role": "编程专家",
    "question": "什么是Python？"
})
print(result.content)
```

**LCEL（LangChain Expression Language）语法：**
- 用 `|` 连接各个组件
- 前一个组件的输出是后一个组件的输入
- 支持链式调用、流式输出、批量处理等

```
输入字典 → prompt → model → 输出AIMessage
```

**链的各种调用方式：**

```python
# 1. 同步调用
result = chain.invoke({"input": "你好"})

# 2. 流式调用（逐字返回）
for chunk in chain.stream({"input": "写一首诗"}):
    print(chunk.content, end="", flush=True)

# 3. 异步调用
result = await chain.ainvoke({"input": "你好"})

# 4. 异步流式
async for chunk in chain.astream({"input": "写一首诗"}):
    print(chunk.content, end="", flush=True)

# 5. 批量调用
results = chain.batch([
    {"input": "问题1"},
    {"input": "问题2"},
])
```

### 第四步：工具（Tools）开发 - 让智能体拥有能力

工具（Tools）是智能体最重要的能力扩展。没有工具，智能体只能用训练数据里的知识回答问题；有了工具，智能体就能调用外部能力。

#### 4.1 什么是工具？

工具就是智能体可以调用的**函数/方法**，用来：
- 🔍 搜索互联网
- 📊 查询数据库
- 🔢 进行数学计算
- 📁 读写文件
- 🌐 调用外部API

#### 4.2 定义工具的三种方式

**方式1：@tool 装饰器（最常用，推荐）**

这是最简单的方式，直接在函数上加 `@tool` 装饰器。函数的docstring会作为工具的描述，LLM会根据描述来决定是否调用这个工具。

例子来自你的项目 [match_engine.py](file:///f:/llm-hint/langchain-2026/deep-agents/fifa-world-cup-2026-v8/tools/match_engine.py#L49-L109)：

```python
from langchain_core.tools import tool

@tool
def tool_calculate_match_probs(home_team: str, away_team: str) -> str:
    """
    计算两队对阵的概率参数，供LLM生成比赛事件时使用。
    返回：控球率、进球概率、各位置进球权重、实力对比等数字参数。
    """
    # ... 具体实现 ...
    return f"""=== 比赛概率参数 ===
主队: {home_team} | 客队: {away_team}
...
"""
```

**关键点：**
- 函数名要清晰（LLM会根据名字理解用途）
- docstring要写清楚（LLM会读这个来判断什么时候调用）
- 参数类型注解要明确（`home_team: str`）
- 返回值最好是字符串（方便LLM理解）

**方式2：StructuredTool（复杂参数时使用）**

当工具需要复杂的参数结构时使用：

```python
from langchain_core.tools import StructuredTool
from pydantic import BaseModel, Field

class SearchInput(BaseModel):
    query: str = Field(description="搜索关键词")
    max_results: int = Field(default=5, description="最多返回多少条结果")

def search_function(query: str, max_results: int = 5):
    """执行搜索并返回结果"""
    # ... 实现搜索逻辑 ...
    return f"搜索 '{query}' 的结果..."

search_tool = StructuredTool.from_function(
    func=search_function,
    name="web_search",
    description="搜索互联网获取实时信息",
    args_schema=SearchInput,
)
```

**方式3：直接继承 BaseTool（最灵活）**

需要完全控制工具行为时使用：

```python
from langchain_core.tools import BaseTool
from typing import Type
from pydantic import BaseModel, Field

class WeatherInput(BaseModel):
    city: str = Field(description="城市名称")

class WeatherTool(BaseTool):
    name: str = "get_weather"
    description: str = "获取指定城市的天气信息"
    args_schema: Type[BaseModel] = WeatherInput

    def _run(self, city: str):
        # 实际获取天气的逻辑
        return f"{city}今天晴，温度25°C"
```

#### 4.3 让智能体使用工具

定义好工具后，需要把工具绑定到模型上，让LLM知道有哪些工具可用。

```python
from langchain_core.tools import tool
from langchain_core.messages import HumanMessage, SystemMessage

# 1. 定义工具
@tool
def get_weather(city: str) -> str:
    """获取指定城市的天气信息"""
    return f"{city}今天晴，25°C"

@tool
def calculator(expression: str) -> str:
    """计算数学表达式，如 '2+3*4' """
    return str(eval(expression))

tools = [get_weather, calculator]

# 2. 绑定工具到模型
model_with_tools = model.bind_tools(tools)

# 3. 调用（LLM会自动判断是否需要调用工具）
messages = [HumanMessage("北京今天天气怎么样？")]
response = model_with_tools.invoke(messages)

print(response.tool_calls)  # 查看LLM想要调用的工具
# 输出可能是: [{'name': 'get_weather', 'args': {'city': '北京'}, 'id': '...'}]
```

#### 4.4 工具调用的完整流程（ReAct模式）

智能体调用工具的标准流程是 **"思考-行动-观察"** 循环：

```
用户问题 → LLM思考 → 决定调用工具 → 执行工具 → 返回结果 → LLM继续思考
     ↑                                                        ↓
     └──────────────── 回答用户问题 ←──────────────────────────┘
```

完整代码示例：

```python
from langchain_core.messages import HumanMessage, AIMessage, ToolMessage

messages = [HumanMessage("北京和上海的温度差多少？")]

# 第一轮：LLM决定调用工具
response = model_with_tools.invoke(messages)
messages.append(response)

# 执行所有工具调用
for tool_call in response.tool_calls:
    tool_name = tool_call["name"]
    tool_args = tool_call["args"]
    tool_id = tool_call["id"]
    
    # 根据工具名找到对应函数并执行
    if tool_name == "get_weather":
        result = get_weather.invoke(tool_args)
    elif tool_name == "calculator":
        result = calculator.invoke(tool_args)
    else:
        result = f"未知工具: {tool_name}"
    
    # 把工具结果返回给LLM
    messages.append(ToolMessage(content=result, tool_call_id=tool_id))

# 第二轮：LLM根据工具结果生成最终回答
final_response = model_with_tools.invoke(messages)
print(final_response.content)
```

#### 4.5 工具设计的最佳实践

从你的FIFA项目中，我们可以学到很好的工具设计原则：

**1. 工具职责单一**
- `tool_calculate_match_probs` → 只算概率
- `tool_get_lineup_for_simulation` → 只拿阵容
- 每个工具做一件事，做好一件事

**2. 工具返回结构化的文本**
- 用清晰的格式（标题、分隔线、键值对）
- 方便LLM理解和提取信息

**3. 工具名 + docstring 清晰明确**
- LLM主要靠名字和描述来判断要不要调用
- 名字要动词开头，描述要讲清楚输入输出

### 第四步：LangGraph状态图 - 构建智能体的核心框架

LangGraph是LangChain推出的状态图框架，用于构建复杂的智能体工作流。它是现代智能体开发的核心。

#### 5.1 为什么需要LangGraph？

普通的Chain是**线性**的：输入 → 处理1 → 处理2 → 输出

而智能体需要**循环和分支**：
- 思考 → 调用工具 → 观察 → 再思考 → ...
- 根据条件走不同的分支
- 出错了可以重试

LangGraph就是用来构建这种**有状态的、有循环的、有分支的**工作流。

#### 5.2 核心概念

| 概念 | 说明 |
|------|------|
| **State（状态）** | 图中流动的数据，所有节点都能读写 |
| **Node（节点）** | 处理单元，接收状态，返回状态的更新 |
| **Edge（边）** | 连接节点，决定执行顺序 |
| **START** | 入口节点 |
| **END** | 结束节点 |
| **Conditional Edge（条件边）** | 根据状态决定走哪个分支 |

#### 5.3 最简单的LangGraph示例：你的聊天机器人

让我们用你项目中的代码来理解，位置：[chain_wrapper/chat.py](file:///f:/llm-hint/langchain-2025/web-project/langserve-api/chain_wrapper/chat.py)

```python
from langgraph.graph import START, MessagesState, StateGraph
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_community.chat_models.tongyi import ChatTongyi

# 1. 定义状态（使用内置的MessagesState）
# MessagesState 自带一个 "messages" 列表字段
workflow = StateGraph(state_schema=MessagesState)

# 2. 定义提示词模板
prompt_template = ChatPromptTemplate.from_messages([
    ("system", "你是一位乐于助人的助手。尽你所能回答所有问题。"),
    MessagesPlaceholder(variable_name="messages"),
])

model = ChatTongyi(streaming=True, name="qwen-turbo")

# 3. 定义节点函数
# 输入：当前状态
# 输出：状态的更新（会自动合并到状态中）
def call_model(state: MessagesState):
    prompt = prompt_template.invoke(state)  # 用当前状态渲染提示词
    response = model.invoke(prompt)         # 调用模型
    return {"messages": response}           # 返回要更新的状态

# 4. 添加节点和边
workflow.add_edge(START, "model")     # START → model 节点
workflow.add_node("model", call_model)  # 添加 model 节点

# 5. 编译图
app = workflow.compile()
```

**这个图的结构很简单：**
```
START → model → END
```

**使用方式：**
```python

    "messages": [HumanMessage("你好！")]
})

# 查看结果

    print(f"{type(msg).__name__}: {msg.content}")
```

#### 5.4 带工具调用的智能体图（ReAct模式）

这是更典型的智能体结构，有一个**循环**：模型 → 判断是否调用工具 → 执行工具 → 回到模型

```python
from langgraph.graph import START, END, StateGraph, MessagesState
from langgraph.prebuilt import ToolNode
from langchain_core.messages import HumanMessage

# 1. 定义工具
@tool
def get_weather(city: str) -> str:
    """获取城市天气"""
    return f"{city}晴，25°C"

tools = [get_weather]
tool_node = ToolNode(tools)  # LangGraph内置的工具执行节点

# 2. 定义模型节点
model_with_tools = model.bind_tools(tools)

def call_model(state: MessagesState):
    response = model_with_tools.invoke(state["messages"])
    return {"messages": [response]}

# 3. 定义条件边函数：判断是否需要继续
def should_continue(state: MessagesState):
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "tools"  # 有工具调用，走工具分支
    return END          # 没有工具调用，结束

# 4. 构建图
workflow = StateGraph(MessagesState)

workflow.add_node("agent", call_model)
workflow.add_node("tools", tool_node)

workflow.add_edge(START, "agent")
workflow.add_conditional_edges(
    "agent",           # 从 agent 节点出发
    should_continue,   # 判断函数
    {
        "tools": "tools",  # 返回 "tools" 就去 tools 节点
        END: END,          # 返回 END 就结束
    }
)
workflow.add_edge("tools", "agent")  # 工具执行完回到 agent（形成循环）

# 5. 编译
app = workflow.compile()
```

**这个图的结构：**
```
START → agent →? → tools → agent →? → ...
                ↓
               END
```

**执行流程：**
1. 用户输入 → agent节点（LLM思考）
2. LLM决定调用工具 → 去tools节点执行
3. 工具结果返回 → 回到agent节点（LLM继续思考）
4. LLM觉得够了 → 结束

#### 5.5 自定义状态State

除了内置的`MessagesState`，你还可以定义自己的状态：

```python
from typing import TypedDict, Annotated, List
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

# 定义状态类型
class AgentState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]  # 对话历史
    user_info: dict                                         # 用户信息
    search_results: list                                    # 搜索结果
    step_count: int                                         # 步数计数

# 使用
workflow = StateGraph(AgentState)
```

**关键点：**
- `Annotated[..., add_messages]` 表示消息列表用追加方式合并
- 普通字段（如 `user_info`）会被直接覆盖
- 节点函数返回的字典会被合并到状态中

#### 5.6 LangGraph的常用调用方式

```python
# 1. 同步调用

# 2. 流式调用（逐事件返回）
for event in app.stream({"messages": [HumanMessage("你好")]}):
    for key, value in event.items():
        print(f"节点 {key} 输出:", value)

# 3. 异步调用

# 4. 异步流式
async for event in app.astream({"messages": [HumanMessage("你好")]}):
    print(event)
```

### 第五步：记忆（Memory）系统 - 让智能体记住对话

没有记忆的智能体每次对话都是"失忆"的，它不记得你之前说过什么。记忆系统让智能体能够记住历史对话。

#### 6.1 记忆的类型

| 记忆类型 | 作用 | 例子 |
|---------|------|------|
| **短期记忆（会话记忆）** | 单次对话内的历史消息 | 你和智能体的聊天记录 |
| **长期记忆** | 跨会话的持久化知识 | 用户偏好、历史总结 |
| **实体记忆** | 提取出的关键实体信息 | 用户名、喜好、重要事实 |

#### 6.2 LangGraph中的记忆：Checkpointer

LangGraph通过 **Checkpointer（检查点）** 机制来实现记忆。

**基本原理：**
- 每次图执行后，状态会被保存到Checkpointer中
- 下次执行时，可以从之前保存的状态继续
- 通过 `thread_id` 来区分不同的会话

**你的项目中的实现（chat.py）：**
位置：[chain_wrapper/chat.py](file:///f:/llm-hint/langchain-2025/web-project/langserve-api/chain_wrapper/chat.py#L33-L40)

```python
from langgraph.checkpoint.memory import MemorySaver

# 1. 创建记忆保存器
memory = MemorySaver()

# 2. 编译图时传入 checkpointer
app = workflow.compile(checkpointer=memory)

# 3. 配置会话ID
config = {
    "configurable": {
        "session_id": time.time(),  # 会话ID
        "thread_id": time.time()    # 线程ID（用于记忆）
    }
}
```

**使用记忆进行多轮对话：**

```python
from langchain_core.messages import HumanMessage

# 第一轮对话
result1 = app.invoke(
    {"messages": [HumanMessage("我叫小明")]},
    config={"configurable": {"thread_id": "user_123"}}
)
print(result1["messages"][-1].content)  # AI的回答

# 第二轮对话（同一个 thread_id，会自动带上历史）
result2 = app.invoke(
    {"messages": [HumanMessage("我叫什么名字？")]},
    config={"configurable": {"thread_id": "user_123"}}
)
print(result2["messages"][-1].content)  # 应该能回答"小明"
```

**关键点：**
- `thread_id` 是会话的唯一标识
- 同一个 `thread_id` 的调用会共享记忆
- `MemorySaver` 是内存存储，程序重启就没了
- 生产环境可以用数据库存储（如 SQLite、PostgreSQL 等）

#### 6.3 不同的Checkpointer后端

| 后端 | 特点 | 适用场景 |
|------|------|---------|
| `MemorySaver` | 内存存储，快但不持久 | 开发测试、临时会话 |
| `SqliteSaver` | SQLite数据库，文件存储 | 小型应用、本地部署 |
| `PostgresSaver` | PostgreSQL数据库 | 生产环境、高并发 |

**使用 SqliteSaver 的例子：**

```python
from langgraph.checkpoint.sqlite import SqliteSaver

# 使用文件存储
memory = SqliteSaver.from_conn_string("checkpoints.sqlite")

app = workflow.compile(checkpointer=memory)
```

你的FIFA项目中使用了更高级的长期记忆方式——通过文件存储的"系统级记忆"。

这种方式的特点：

#### 6.5 记忆的最佳实践

1. **短期记忆用Checkpointer**
   - 自动管理对话历史
   - 通过thread_id区分用户

2. **长期记忆用文件/数据库**
   - 保存用户画像、重要事实
   - 定期从对话中提取关键信息

3. **注意上下文窗口限制**
   - 对话太长会超过LLM的上下文窗口
   - 需要时进行摘要压缩

4. **记忆安全**
   - 敏感信息不要存在记忆里
   - 不同用户的记忆要严格隔离

### 第六步：技能（Skills）与上下文工程 - 提升智能体能力

这是高级智能体的核心概念。你的FIFA世界杯项目就是一个很好的例子。

#### 7.1 什么是技能（Skills）？

技能（Skill）是给LLM的一份**"行为说明书"**，告诉LLM在特定场景下：
- 应该按什么步骤执行
- 应该调用哪些工具
- 应该输出什么格式
- 有什么约束条件

你项目中的例子：[match_simulation/SKILL.md](file:///f:/llm-hint/langchain-2026/deep-agents/fifa-world-cup-2026-v8/skills/match_simulation/SKILL.md)

```markdown
# Match Simulation Skill — LLM行为指南

## 执行步骤（必须严格遵循）
### Step 1: 数据收集（tool调用阶段）
调用顺序（必须全部执行）:
1. tool_get_squad(home_team)        ← 获取主队26人名单
2. tool_get_squad(away_team)        ← 获取客队26人名单
3. tool_search_squad_news(home)     ← 【联网】获取主队最新伤病/首发
...

### Step 3: FM风格比赛实况（流式核心）
**关键规则**:
- 按时间顺序推进，每个事件单独一行
- 每5-10分钟产生1-2个事件
- 事件类型及概率权重...
```

#### 7.2 技能的工作原理

```

    ↓
作为系统提示词的一部分注入给LLM
    ↓
LLM读取并理解"我应该怎么做"
    ↓
按照技能描述的步骤调用工具、生成输出
```

#### 7.3 上下文工程（Context Engineering）

上下文工程是指**如何高效地给LLM提供正确的上下文信息**。这是提升智能体能力的关键。

你的FIFA项目展示了5大上下文工程模式：

| 模式 | 说明 | 项目中的位置 |
|------|------|-------------|
| **Input Context** | 按需加载数据，只给需要的信息 | `data_loader.py` 的 `get_squad()` |
| **Runtime Context** | 运行时动态注入外部信息 | `search_tools.py` 的联网搜索 |
| **Context Compression** | 压缩信息，只保留关键内容 | 工具返回结构化数据，LLM看摘要 |
| **Context Isolation** | 子智能体只看自己的数据，不继承全局 | `match_subagent.py` |
| **Long-term Memory** | 跨会话的持久化记忆 | `memories/AGENTS.md` |

让我们逐个理解：

**1. Input Context（输入上下文）**
```python
# ❌ 不好的做法：把所有数据都给LLM
all_teams_data = load_all_1222_players()  # 太多了！

# ✅ 好的做法：只给LLM需要的数据
home_squad = get_squad(home_team)   # 只要主队26人
away_squad = get_squad(away_team)   # 只要客队26人
```

**2. Runtime Context（运行时上下文）**
```python
# 静态数据（本地文件）
squad = get_squad("巴西")

# 动态数据（运行时获取）
injury_news = tool_search_squad_news("巴西")  # 实时伤病
```

**3. Context Compression（上下文压缩）**
```python
# ❌ 原始搜索结果：10000字的网页内容
raw_html = fetch_webpage(url)

# ✅ 压缩后：结构化的关键信息
result = {
    "injured_players": ["内马尔", "卡塞米罗"],
    "return_date": "2026-06-15",
    "impact": "medium"
}
```

**4. Context Isolation（上下文隔离）**
```
Supervisor（全局上下文）
    ↓ 调用子智能体时，只传递必要信息
Match SubAgent（只看到这场比赛的数据）
    ↓
比赛模拟完成，返回结构化结果
```

**5. Long-term Memory（长期记忆）**
```
memories/AGENTS.md  →  系统级长期记忆
  - 智能体身份
  - 核心能力
  - 重要约束
  - 模型参数
```

#### 7.4 技能设计的最佳实践

从你的项目中可以学到：

1. **明确的步骤**
   - 用 Step 1, Step 2, Step 3 清晰列出
   - 每个步骤做什么、调用什么工具

2. **明确的输出格式**
   - 给LLM输出的模板
   - 用代码块或分隔线明确格式

3. **明确的约束**
   - 什么可以做，什么不可以做
   - 比如"门将绝对不能进球"

4. **技能 + 工具配合**
   ```
   SKILL.md = 告诉LLM"怎么做"
   tools.py = 提供"做的能力"
   ```

### 第七步：多智能体（Multi-Agent）系统架构

当任务变得复杂时，单个智能体可能不够用。这时候就需要多个智能体协作，各司其职。

#### 8.1 为什么需要多智能体？

- **专业分工**：每个智能体专注于自己擅长的领域
- **上下文隔离**：每个智能体只看自己需要的信息
- **并行处理**：多个智能体可以同时工作，提高效率
- **可扩展性**：新增功能只需添加新的智能体

#### 8.2 常见的多智能体架构

**1. 主管-下属（Supervisor-Worker）模式**
```
         ┌──────────────┐
         │  Supervisor  │  ← 主管智能体（分配任务、汇总结果）
         └──────┬───────┘
         ┌──────┴───────┐
    ┌────┴────┐   ┌─────┴────┐
    │ Worker1 │   │ Worker2  │  ← 子智能体（执行具体任务）
    └─────────┘   └──────────┘
```

这是你的FIFA项目采用的架构：
- `WorldCupSupervisor` → 主管，负责路由和调度
- `match_subagent` → 子智能体，负责单场比赛模拟
- `preview_subagent` → 子智能体，负责赛前分析

**2. 流水线（Pipeline）模式**
```
用户输入 → 意图识别Agent → 信息检索Agent → 回答生成Agent → 输出
```

**3. 委员会（Committee）模式**
```
         ┌──────────────┐
         │  汇总Agent   │
         └──────┬───────┘
    ┌─────────┬─┴─┬─────────┐
┌───┴───┐ ┌───┴──┐ ┌────────┴───┐
│专家A  │ │专家B  │ │  专家C     │
└───────┘ └──────┘ └────────────┘
```

```

  ├── 只能调用比赛相关的工具
  └── 只能读取比赛模拟的技能
       ↓
返回结构化结果（比分、统计数据）
```

#### 8.5 多智能体开发的注意事项

1. **明确职责边界**
   - 每个智能体做什么、不做什么，要清晰
   - Supervisor只做调度，不做具体业务

2. **统一接口规范**
   - 子智能体的输入输出格式要统一
   - 方便Supervisor调用和结果整合

3. **错误处理和降级**
   - 子智能体失败了怎么办？
   - 你的FIFA项目中有很好的fallback机制：
     - LLM不可用时用规则引擎兜底
     - 搜索失败时用DDG降级

4. **资源共享**
   - LLM实例、配置等可以共享
   - 但数据上下文要隔离

### 第八步：API服务化 - LangServe与FastAPI部署

智能体开发完成后，需要把它部署成API服务，让前端或其他系统能够调用。

#### 9.1 两种部署方式

你的项目中展示了两种方式：

#### 9.2 方式1：LangServe 自动部署

LangServe可以自动把LangChain的Runnable（Chain、Agent等）部署成API。

**你的项目中的例子：**
位置：[main.py](file:///f:/llm-hint/langchain-2025/web-project/langserve-api/main.py#L36-L52)

```python
from fastapi import FastAPI
from langserve import add_routes
from langchain.prompts import ChatPromptTemplate
from langchain_community.chat_models.tongyi import ChatTongyi

# 1. 创建Chain
prompt_template = ChatPromptTemplate.from_messages([
    ("system", "Translate the following into {language}:"),
    ("user", "{text}")
])
model = ChatTongyi(streaming=True)
chain = prompt_template | model

# 2. 创建FastAPI应用
app = FastAPI(
    title="LangChain Server",
    version="1.0",
    description="A simple api server using Langchain's Runnable interfaces",
)

# 3. 用LangServe添加路由
add_routes(
    app,
    chain,
    path="/chain",          # 路由路径
)

# 4. 启动
if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8000)
```

**LangServe提供的功能：**
- `POST /chain/invoke` → 同步调用
- `POST /chain/stream` → 流式调用
- `POST /chain/stream_log` → 流式日志（含中间步骤）
- `GET /chain/playground` → 在线Playground界面

**优缺点：**
- ✅ 简单，几行代码就搞定
- ✅ 自带Playground，方便测试
- ❌ 定制化程度有限
- ❌ 不适合复杂的LangGraph智能体

#### 9.3 方式2：FastAPI 自定义接口（推荐）

对于复杂的智能体（特别是LangGraph），建议用FastAPI自己写接口。

**你的项目中的例子：**
位置：[router_api.py](file:///f:/llm-hint/langchain-2025/web-project/langserve-api/router_api.py)

```python
from fastapi import APIRouter
from starlette.responses import StreamingResponse
from pydantic import BaseModel
from langchain_core.messages import AIMessageChunk, HumanMessage
from chain_wrapper import chat as chat_chain

router = APIRouter()

# 定义请求体
class Item(BaseModel):
    content: str

# 流式生成函数
async def generate_response(content, type):
    input_messages = [HumanMessage(content)]
    
    # LangGraph 流式输出
    async for message_chunk, metadata in chat_chain.app.astream(
        {"messages": input_messages},
        config=chat_chain.config,
        stream_mode="messages",
    ):
        # 处理消息块
        if isinstance(metadata, AIMessageChunk):
            message_str = str(message_chunk.content)
        else:
            message_str = message_chunk.content
        
        # 编码为字节流
        yield message_str.encode('utf-8')

# 聊天接口
@router.post("/api/chat")
async def chat(item: Item):
    return StreamingResponse(
        generate_response(item.content, "standard"),
        media_type="text/event-stream"
    )
```

#### 9.4 流式输出（SSE）

智能体通常需要流式输出（逐字返回），这需要用SSE（Server-Sent Events）。

**FastAPI中实现SSE的两种方式：**

**方式1：StreamingResponse（简单）**
```python
from fastapi import FastAPI
from starlette.responses import StreamingResponse

app = FastAPI()

async def generate():
    for i in range(10):
        yield f"第{i}条消息\n".encode('utf-8')
        await asyncio.sleep(0.5)

@app.get("/stream")
async def stream():
    return StreamingResponse(
        generate(),
        media_type="text/event-stream"
    )
```

**方式2：标准SSE格式（推荐）**
```python
async def generate_sse():
    for i in range(10):
        # SSE格式：data: {内容}\n\n
        yield f"data: 第{i}条消息\n\n".encode('utf-8')
        await asyncio.sleep(0.5)
```

**前端接收SSE的方式：**
```javascript
// 使用 EventSource
const eventSource = new EventSource('/stream');
eventSource.onmessage = (event) => {
    console.log('收到:', event.data);
};

// 或者使用 fetch + ReadableStream
const response = await fetch('/stream');
const reader = response.body.getReader();
while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    console.log('收到:', new TextDecoder().decode(value));
}
```

#### 10.3 核心流程分析：单场比赛模拟

让我们跟踪一次"单场比赛模拟"的完整调用链：

```
用户请求（前端/CLI/API）
    ↓
api/server.py 接收请求
    ↓
supervisor_agent.py → dispatch_single_match()
    ↓ 判断LLM是否可用
    ├─ LLM可用 → match_subagent.py → run_llm_match()
    └─ LLM不可用 → match_agent.py → _stream_fallback()（规则引擎）
    ↓
返回SSE流式结果
```

**LLM模式下的详细流程：**

```
1. 加载技能（SKILL.md）→ 告诉LLM比赛模拟的步骤
2. 调用工具收集数据：
   - tool_get_squad(主队) → 获取阵容
   - tool_get_squad(客队) → 获取阵容
   - tool_search_squad_news(主队) → 联网查伤病
   - tool_search_squad_news(客队) → 联网查伤病
   - tool_get_tactics(两队) → 获取战术
   - tool_calculate_match_probs() → 计算概率参数
3. LLM生成流式解说（FM风格）
4. 输出赛后统计报告
```

#### 10.4 这个项目的亮点设计

**亮点1：优雅的降级策略（Fallback）**

- LLM不可用时，自动切换到规则引擎
- 搜索工具也有多层降级：Tavily → DDG → 本地数据
- 保证系统在任何情况下都能用

**亮点2：上下文工程的5种模式全部用上**

- Input Context：按需加载数据
- Runtime Context：联网搜索注入
- Context Compression：工具返回结构化数据
- Context Isolation：子智能体独立上下文
- Long-term Memory：AGENTS.md 长期记忆

**亮点3：清晰的分层架构**

- 每一层职责明确
- 业务逻辑不分散
- 易于维护和扩展

**亮点4：并发支持**

- 小组比赛可以并发模拟
- 用 `asyncio` 实现异步并发
- 支持配置并发数

#### 10.5 从这个项目中可以学到什么

1. **先想清楚架构再写代码**
   - 不是上来就写prompt
   - 而是先规划：有哪些智能体？各自负责什么？怎么协作？

2. **工具和技能要分开**
   - 工具是"能力"，技能是"怎么用能力"
   - 这样LLM才能更好地使用工具

3. **一定要有降级方案**
   - LLM可能挂、API可能限流、网络可能不通
   - 设计系统时要考虑这些情况

4. **数据驱动，而不是只靠LLM**
   - 能用规则和数据解决的，就不要让LLM算
   - LLM负责"创造性"的部分（如解说文字）
   - 计算部分用代码实现（如概率计算）
